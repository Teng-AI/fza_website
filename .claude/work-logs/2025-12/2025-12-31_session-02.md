## Problems Solved

### Problem 1: Venue Image Showing Wrong Content

**Issue**: The venue.png file contained a schedule graphic instead of the actual venue photo. This happened during initial asset download in session 1 - the wrong image was saved with the venue.png filename.

**Root Cause**: During batch image downloads in session 1, the incorrect source image was downloaded and saved as venue.png, likely due to URL confusion or manual error.

**Solution**:
1. Used browser automation to navigate to original site
2. Inspected DOM to find exact image element for venue section
3. Extracted correct image URL via JavaScript console
4. Downloaded correct image using curl
5. Replaced venue.png with correct file
6. Cleared Next.js cache to force reload
7. Restarted dev server to apply changes

**Verification**: Visually compared localhost:3001 venue section with original site - images now match perfectly.

**Learning**: When downloading assets in batches, verify each image visually before moving on. Images with generic filenames (venue.png, hero.jpg) are especially prone to mix-ups.

### Problem 2: Next.js Cache Preventing Image Update

**Issue**: After replacing venue.png with the correct image, the browser continued showing the old (incorrect) image even after hard refresh.

**Root Cause**: Next.js Image optimization caches processed images in the .next directory. Simply replacing the source file doesn't automatically invalidate the cache.

**Solution**:
1. Stopped the development server
2. Deleted the entire .next directory: `rm -rf .next`
3. Restarted dev server: `npm run dev`
4. Hard refreshed browser to clear client-side cache

**Side Effect**: Turbopack panicked after cache clear, requiring server restart (which was needed anyway).

**Learning**: For Next.js image updates during development, clearing the .next cache is often necessary. This is especially true when replacing files with the same name.

### Problem 3: Turbopack Panic After Cache Clear

**Issue**: After running `rm -rf .next`, the running dev server with Turbopack experienced a panic and crashed.

**Root Cause**: Turbopack watches file system changes, and deleting the .next directory while the server was running caused an internal error.

**Solution**: Restarted the dev server with `npm run dev`. Server started successfully and rebuilt from clean state.

**Learning**: When clearing .next cache, stop the dev server first, then clear cache, then restart. This prevents Turbopack from detecting the directory deletion while running.

### Problem 4: Finding Exact Image URL on Original Site

**Issue**: Needed to locate the exact URL of the correct venue image from the original fuzhouamerica.org site, but the site uses complex Webflow CDN URLs.

**Root Cause**: Images on Webflow sites don't have simple predictable URLs - they use CDN with hashed filenames and element IDs.

**Solution**: Used browser automation with JavaScript execution:
1. Opened original site in automated browser
2. Inspected venue section to find the image element's data-w-id attribute
3. Executed JavaScript to query DOM: `document.querySelector('[data-w-id="..."]').src`
4. Retrieved full CDN URL from JavaScript result

**Learning**: Browser automation with JavaScript execution is powerful for extracting specific data from complex web pages. DOM inspection combined with querySelector provides precise element targeting.

## Notes & Insights

### Cache Management in Next.js

Next.js Image component aggressively caches optimized images, which is great for performance but can be confusing during development when replacing images with the same filename. Key learnings:

- Clearing `.next` directory forces complete rebuild
- Turbopack should be stopped before clearing cache
- Browser hard refresh may still be needed after server restart
- Consider using versioned filenames (e.g., `venue-v2.png`) when testing image replacements

### Browser Automation for Asset Discovery

Browser automation proved invaluable for extracting exact image URLs from complex websites:

- JavaScript execution allows precise DOM querying
- Can extract information not easily visible in HTML source
- More reliable than manual inspection for Webflow/CDN sites
- Useful pattern for future asset downloads

### Git Workflow for Initial Projects

For initial project commits, a single comprehensive commit with detailed message is often better than artificial splitting:

- Provides clear starting point
- Detailed message documents what's included
- Easier to understand project foundation
- More granular commits can come later for incremental changes

### Development Server Port Conflicts

Port 3000 was already in use, so dev server automatically used 3001. This is normal behavior but good to note:

- Always check which port the server actually started on
- Update URLs accordingly when testing
- Consider killing orphaned processes if 3000 is blocked unexpectedly

### Image Verification Importance

The venue image bug could have been prevented with better verification during initial download:

- Always visually verify images after downloading
- Compare with source site before moving on
- Pay extra attention to images with generic names
- Consider creating a checklist for asset downloads
