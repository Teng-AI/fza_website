## Key Decisions Made

### Decision 1: Hero Section Height Reduction

**Context**: Original design used full viewport height (h-screen) for hero sections, which pushed content below the fold on most screens.

**Options considered:**
1. Keep h-screen for dramatic impact
2. Reduce to h-[60vh] for moderate height
3. Reduce to h-[50vh] for compact design

**Decision**: Chose h-[50vh] (50% of viewport height)

**Rationale**: The 50vh height provides adequate visual impact while ensuring that key content (page titles, CTAs) and the beginning of body content are visible without scrolling. This improves engagement and reduces bounce rates, especially on mobile devices. The rounded bottom corners visually separate the hero from content sections.

### Decision 2: Navigation Typography Changes

**Context**: Needed to match the visual style of the original fuzhouamerica.org while maintaining readability.

**Options considered:**
1. Keep current mixed-case with regular spacing
2. ALL CAPS with standard spacing
3. ALL CAPS with increased letter-spacing (tracking-wide)

**Decision**: ALL CAPS with tracking-wide and text-sm

**Rationale**: ALL CAPS provides a bold, confident appearance appropriate for a nonprofit organization. The increased letter-spacing (tracking-wide) prevents the ALL CAPS text from appearing cramped or aggressive. The smaller text size (text-sm) balances the visual weight of the uppercase letters and maintains hierarchy with the logo.

### Decision 3: Button Color Change for "JOIN OUR COMMUNITY"

**Context**: Original design used red (bg-red-600) for the primary CTA button in the header.

**Options considered:**
1. Keep red for consistency with other accent colors
2. Change to black for professional appearance
3. Use brand color if one exists

**Decision**: Changed to black (bg-black)

**Rationale**: Black provides better contrast against the white header background and appears more sophisticated. Red buttons can sometimes appear too aggressive or "salesy" for a community-focused nonprofit. Black maintains strong visual hierarchy while feeling more welcoming and professional.

### Decision 4: Conference Schedule Layout Redesign

**Context**: Previous accordion/FAQ-style layout made it difficult to see the full schedule at a glance.

**Options considered:**
1. Keep accordion layout with improvements
2. Switch to table layout
3. Implement timeline layout with visual indicators

**Decision**: Timeline layout with date pills, time indicators, and event cards

**Rationale**: The timeline layout provides better scanability and allows users to see the full conference schedule without clicking through accordions. Visual elements (colored date pills, time stamps) create clear day boundaries and event groupings. This design pattern is commonly used for event schedules and feels familiar to users.

### Decision 5: Production Deployment Platform

**Context**: Needed to deploy the site to a production environment for stakeholder review and testing.

**Options considered:**
1. Deploy to Netlify
2. Deploy to Vercel
3. Deploy to custom server/hosting

**Decision**: Vercel with continuous deployment from GitHub

**Rationale**: Vercel is the creator of Next.js and provides optimal performance and developer experience for Next.js applications. The platform offers automatic deployments on git push, preview deployments for branches, and excellent Next.js integration. Free tier is sufficient for the project's current needs.

### Decision 6: Git Workflow for Production

**Context**: Site is now in production and requires more careful change management.

**Options considered:**
1. Continue committing directly to main branch
2. Use feature branches + pull requests for all changes
3. Hybrid approach: direct commits during dev, branches in production

**Decision**: Use feature branches + pull requests going forward

**Rationale**: Now that the site is deployed to production, changes need more careful review and testing. Feature branches allow for isolated development and testing before merging to main. Pull requests provide opportunity for code review and documentation of changes. This prevents accidental breaking changes from reaching production users.

## Problems Solved

### Problem 1: Build Verification Before Deployment

**Issue**: Needed to ensure no TypeScript errors or build issues existed before deploying to production.

**Solution**: Ran `npm run build` to execute production build process, which validates all TypeScript types, catches unused imports, and builds optimized production assets.

**Learning**: Always run production build before deploying to catch errors that might not appear in development mode. Next.js production build is more strict than dev mode.

### Problem 2: Vercel CLI Authentication

**Issue**: First-time deployment required authentication with Vercel platform.

**Solution**: Used `vercel` command which automatically triggered browser-based OAuth flow with GitHub. This linked the Vercel account, GitHub account, and project repository.

**Learning**: Vercel CLI provides smooth onboarding experience. GitHub authentication enables automatic repository connection for continuous deployment.

### Problem 3: Conference Schedule Responsive Design

**Issue**: Timeline layout needed to work on both desktop and mobile screens without losing readability.

**Solution**: Implemented responsive grid layout using Tailwind's responsive utilities. On mobile, schedule stacks vertically with full-width cards. On desktop, uses two-column grid for optimal space usage.

**Learning**: Planning responsive design during initial implementation saves refactoring time. Testing on multiple viewport sizes during development catches layout issues early.
