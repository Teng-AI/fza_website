## Key Decisions Made

### Decision 1: Use Brand Red for All Primary CTAs

**Context**: The site had mixed button colors - some CTAs used black (`bg-foreground`), others used red (`bg-primary`). This created visual inconsistency.

**Options considered:**
1. Keep mixed colors (black for header, red for body CTAs)
2. Standardize all to black for minimalist look
3. Standardize all primary CTAs to brand red

**Decision**: Standardize all primary CTAs to brand red (`bg-primary`)

**Rationale**:
- Brand red is more attention-grabbing for conversion actions
- Creates stronger visual consistency and brand identity
- Aligns with the FZA logo color scheme
- Better hierarchy: red = primary action, subtle colors = secondary actions

### Decision 2: Supabase for Membership Backend

**Context**: Need authentication, member database, admin tools, and event management system. Considered various backend options.

**Options considered:**
1. Firebase (Auth + Firestore)
2. Auth0 (Auth only) + separate database
3. Supabase (Auth + PostgreSQL)
4. Custom Next.js API routes + database

**Decision**: Use Supabase (Auth + PostgreSQL)

**Rationale**:
- Built-in Google OAuth with minimal configuration
- PostgreSQL database with powerful Row Level Security (RLS)
- Real-time subscriptions if needed for live event updates
- Better SQL query capabilities vs NoSQL
- Free tier sufficient for FZA's expected member volume
- Excellent Next.js integration with @supabase/ssr
- Single platform for auth + database reduces complexity

### Decision 3: Auto-Create Member Profiles via Database Trigger

**Context**: Need to capture member data from Google OAuth and allow phone number collection.

**Options considered:**
1. Create profile manually in Next.js API route after signup
2. Use database trigger to auto-create profile from auth.users
3. Require users to fill profile form after first login

**Decision**: Auto-create profile via PostgreSQL trigger, then collect phone later

**Rationale**:
- Database trigger ensures profile always exists when user exists
- Reduces application code complexity
- Atomic operation prevents race conditions
- Users can add phone number after signup via profile page
- Clean separation: Supabase handles auth, trigger handles profile creation

### Decision 4: Middleware-Based Route Protection

**Context**: Need to protect /members/* and /admin/* routes from unauthorized access.

**Options considered:**
1. Check auth in each page component (client-side)
2. Check auth in layout.tsx (server-side)
3. Use Next.js middleware (edge runtime)
4. Supabase RLS only (no route protection)

**Decision**: Use Next.js middleware for route protection

**Rationale**:
- Runs on edge before page even renders (most efficient)
- Single point of control for all protected routes
- Prevents unauthorized users from accessing page code at all
- Complements Supabase RLS (middleware = route, RLS = data)
- Standard Next.js pattern for auth protection

## Problems Solved

### Problem 1: Footer Too Minimal

**Issue**: Original footer only showed copyright text, missing opportunity to provide navigation and contact info.

**Solution**: Redesigned footer with 3-column responsive layout including organization info, navigation links, and complete contact details.

**Learning**: Footers are valuable real estate for SEO, navigation, and establishing organizational legitimacy. Including 501(c)(3) status builds trust.

### Problem 2: Inconsistent CTA Styling

**Issue**: Primary CTAs used different colors (black vs red) across the site, creating visual confusion about what actions were most important.

**Solution**: Audited all buttons and standardized primary CTAs to use `bg-primary` (brand red) consistently.

**Learning**: Color consistency in UI elements directly impacts conversion rates. Users learn to associate specific colors with specific action types.

### Problem 3: Static Images Lack Engagement

**Issue**: All images were static with no interactive feedback, making the site feel less modern.

**Solution**: Added consistent hover effects (scale, shadow, transform) using Tailwind utilities across all pages.

**Learning**: Small interactive details significantly improve perceived quality and user engagement. The key is keeping animations subtle (300ms duration, small scale changes).

## Notes & Insights

### Design Insights

- **Small details matter**: Hover effects and consistent button colors significantly improve perceived quality without major development effort
- **Footer real estate**: The footer is valuable for navigation, SEO, contact info, and trust signals (501(c)(3))
- **Color psychology**: Red CTAs perform better for conversion actions vs neutral colors
- **Responsive design**: All improvements maintained mobile responsiveness using Tailwind breakpoints

### Technical Insights

- **Tailwind utilities**: The combination of `transition-transform duration-300 hover:scale-105 hover:shadow-xl` creates polished hover effects with minimal code
- **Next.js Image component**: Automatically optimizes images while supporting CSS transforms
- **Vercel deployment**: GitHub integration provides seamless CI/CD - push to main triggers automatic deployment
- **Component organization**: Keeping component files focused (Header, Footer) makes updates easier

### Project Management Insights

- **Plan-first approach**: Creating detailed implementation plan before coding saves time and reduces mistakes
- **Quick wins**: Identifying and implementing high-impact, low-effort improvements provides immediate value
- **Iterative improvement**: Ship small improvements frequently rather than waiting for perfect solution
- **Documentation**: Comprehensive planning documents (like the membership system plan) enable async work

### Membership System Planning Insights

- **Complexity estimation**: What seems like "just add login" actually involves 20+ new files and multiple systems
- **Database design**: Auto-creating profiles via triggers is more reliable than application code
- **Security layers**: Middleware + RLS provides defense in depth for protected content
- **Supabase advantages**: Built-in auth + database + RLS reduces boilerplate compared to separate services
